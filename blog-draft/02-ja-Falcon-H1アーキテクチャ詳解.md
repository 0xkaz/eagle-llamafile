# Falcon-H1：数学で32Bを超える7Bモデル

*個人リサーチ / 0xkaz — 2026年3月*

---

Falcon-H1R-7BはAIME 2025で83.1%を記録している。参考までに、Apriel-15Bは82.7%、OLMo 3 Think-32Bは73.7%だ。

7Bモデルが15Bや32Bモデルを数学ベンチマークで打ち負かすのは十分珍しい話で、アーキテクチャを深く見る価値がある。

---

## コアの革新：並列ハイブリッドレイヤー

ほとんどのハイブリッドモデル——AI21 LabsのJambaなど——はAttentionとSSMレイヤーを順番に交互に並べる：

```
Jamba（逐次ハイブリッド）:
  Block 1: Attention
  Block 2: Mamba
  Block 3: Attention
  ...
```

Falcon-H1は異なるアプローチを取る。各ブロック内で、AttentionヘッドとSSMヘッドが**並列に実行**され、出力が結合される：

```
Falcon-H1（並列ハイブリッド）:
  Block N:
    ├── [Attentionヘッド] ──┐
    │                       ├── concat → 出力
    └── [SSMヘッド     ] ──┘
```

小さな実装の違いのように見えるが、意味は大きい：

- AttentionとSSMの比率をモデルサイズごとに独立して調整できる
- 交互配置のオーバーヘッドがない——両メカニズムが同じ入力を受け取る
- Falcon-H1-7Bでは比率はおよそ1:8（Attention:SSM）——SSM優位

---

## 推論にSSMが重要な理由

Transformerには長いコンテキストにおける根本的なスケーリング問題がある：KVキャッシュがシーケンス長の二乗で増大する。128Kトークンのコンテキストでは巨大になる。

SSM（Falcon-H1では特にMamba-2）は代わりに固定サイズの再帰的な状態を使用する：

| | メモリスケーリング | 128Kトークンコンテキスト |
|--|--|--|
| Transformer KVキャッシュ | O(n²) | ギガバイト単位 |
| Mamba-2状態 | O(1) | 固定、小さい |

実測値：Falcon-H1-34B対Qwen2.5-32B（128K+トークン）：

- **入力スループット: 4倍高速**
- **出力スループット: 8倍高速**

短いコンテキスト（1K以下）ではTransformerがわずかに速い。クロスオーバーポイントは4K〜8K tokens。それ以降、SSMハイブリッドの優位性は指数的に拡大する。

プロダクト開発者向けに言えば：長文書、長いチャット履歴、大きなコードベースを扱うユースケースでは、メモリ効率の差は実質的な意味を持つ。

## パラメータ効率の話

Falcon-H1R-7Bのベンチマーク数値：

| ベンチマーク | H1R-7B | 次の競合 | そのサイズ |
|-------------|--------|---------|-----------|
| AIME 2025 | **83.1%** | Apriel | 15B |
| Mathematics | **73.96%** | Qwen3 | 32B |
| Code & Agentic | **33.95%** | Qwen3 | 32B |
| 推論速度 | ~1,500 tok/s/GPU (batch 64) | Qwen3-8B | ~750 tok/s |

34Bモデルの比較も同様に際立っている：Qwen2.5-72BやLlama3.3-70Bと同等の性能を、おおよそ半分のパラメータ数で実現している。

この効率の理由については議論があるが、主な仮説は：SSMレイヤーは少ないパラメータで長いシーケンスのパターンマッチングが得意で、Attentionレイヤーはローカルなコンテキストと特定の事実の想起を担当する。並列の組み合わせが両方の強みを同時に捉える、というものだ。

## バリアント一覧

Falcon-H1ファミリーは他のモデルファミリーよりも多くのバリアントを持つ：

| モデル | 特徴 |
|--------|------|
| H1-0.5B / 1.5B / 3B / 7B / 34B | 標準サイズ |
| **H1-1.5B-Deep** | 幅を削り深さを増加——7B-10B相当の性能 |
| **H1R-7B** | 推論特化、拡張思考、数学最適化 |
| **H1-Arabic-3B / 7B / 34B** | アラビア語特化（同じハイブリッドアーキ） |
| H1-Tiny (0.6B / 0.09B) | 多言語エッジモデル |

1.5B-Deepバリアントは注目に値する：TIIは1.5Bモデルから7B-10B相当の性能を報告している。もし正確なら、エッジ/モバイルデプロイにとって最も興味深いサイズだ。

アラビア語バリアントは同じ並列ハイブリッドアーキテクチャを使用している。GGUFが公開されれば（2026年3月時点では未公開）、世界初のアラビア語SSMハイブリッドllamafileになる。

## ローカルで実行する

現実的な制限として：Mamba-2のMetal GPUカーネルはM5/A19チップ以降が必要。M4 Proおよびそれ以前では：

```
ggml_metal_device_init: tensor API disabled for pre-M5 and pre-A19 devices
```

CPU推論は問題なく動く。M4 Proでは7B CPUは実用的；34BはCPUで遅いが機能はする。M5 Macがllamafileサポートとともにリリースされれば、長コンテキスト効率を考えると、このモデルのGPU高速化バージョンは本当に印象的なものになるだろう。

```bash
# Falcon-H1R-7Bをローカルで実行（CPU）
./falcon-h1r-7b-Q4_K_M.llamafile \
  -m /zip/Falcon-H1R-7B-Q4_K_M.gguf \
  --cli -p "Solve: if x² + 5x + 6 = 0, find x" \
  -n 200 --nothink
```

## このアーキテクチャが重要だと思う理由

ほとんどの「効率的な」モデルは一つの次元で効率的だ——より小さなサイズ、より速いトレーニング、より低いメモリ。Falcon-H1がデプロイされたプロダクトにとって特に重要な効率性を持つのは：長コンテキストの推論スループットだ。

これが直接効いてくるユースケース：
- 長い会話履歴を持つチャットアプリ
- 大きなPDFやコードベースに対するドキュメントQA
- ツール呼び出し全体にわたって長いコンテキストを維持するエージェント
- 多言語アプリ（アラビア語・日本語含む18言語ネイティブ対応）

APIコストゼロでラップトップで動くローカル推論の代替として、Falcon-H1R-7Bの性能/サイズのトレードオフは現時点で本当に打ち破りにくい。
